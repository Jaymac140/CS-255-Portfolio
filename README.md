CS 255 – DriverPass System Analysis and Design

Project Overview

The DriverPass project involved designing a system for a driving instruction company that wanted to improve how students prepare for their driving exams. The client, DriverPass, needed a web-based system that would allow students to register, schedule lessons, purchase packages, take practice exams, and manage reservations. The system also needed administrative functionality for instructors and managers, along with secure payment processing and audit logging.

The goal of this project was to gather business requirements and translate them into a structured system design using UML models and technical documentation.

⸻

What I Did Particularly Well

I believe I did particularly well in translating the client’s business requirements into clear system models. I carefully reviewed the interview transcript and identified both functional and nonfunctional requirements. I ensured that each UML diagram — use case, activity, sequence, and class — aligned directly with the client’s needs. I also maintained consistency across diagrams so that the system behavior and data structure were logically connected.

⸻

What I Would Revise

If I were to revise one part of this project, I would further refine the visual layout and formatting of some of the UML diagrams to enhance clarity and presentation quality. While the system logic is accurate and complete, improving visual alignment and spacing would make the documentation even more polished for a professional audience.

⸻

Interpreting User Needs

To interpret the user’s needs, I focused on identifying the core problems the client wanted to solve: improving lesson scheduling, preventing double bookings, managing payments securely, and tracking system activity through audit logs. I translated these needs into specific system functions and incorporated them into the design models.

Considering user needs is critical because a technically correct system can still fail if it does not align with how users actually work. Designing with the user in mind ensures usability, reliability, and long-term adoption.

⸻

My Approach to Designing Software

When designing software, I begin by clearly defining stakeholders and system boundaries. I identify functional requirements (what the system must do) and nonfunctional requirements (how the system must perform). I then use modeling techniques such as UML diagrams to visualize workflows, system interactions, and data structures before implementation begins.

In the future, I plan to continue using structured analysis, iterative refinement, and strong documentation practices to ensure systems are scalable, secure, and aligned with business objectives.
